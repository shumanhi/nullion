# Nullion Agent Instructions

## Routing Rules

- Do not infer user intent from hardcoded natural-language words, phrases, regexes, or synonym lists.
- Do not split one user message into multiple tasks based on conjunctions or sentence wording.
- Treat a user message as one request unless there is a structured product signal proving otherwise.
- Allowed signals: explicit UI action, slash/operator command, attachment metadata, file extension, URL/domain, tool result schema, task/frame state, approval state, artifact descriptor, or model-produced structured plan.
- If intent classification is needed, use a model-produced structured output with a schema, then validate it against runtime evidence.
- Prefer Deep Agents, LangGraph, and LangChain for stateful workflows, typed routing, retries, artifact verification, task state, and tool orchestration.
- Deep Agents may receive structured tasks only; do not dispatch them from keyword matching.
- LangGraph nodes should branch on typed state and verified tool/runtime facts, not free-form text phrases.
- LangChain/tool adapters should expose structured tool metadata and outputs so downstream code avoids parsing prose.
- User-facing delivery must treat tool outputs as evidence, not final copy. Render structured tool results through schema-aware presenters that produce concise outcomes, receipts, artifacts, next actions, and safety cautions; do not expose raw JSON, internal records, command dumps, descriptor files, or provider prose unless the user explicitly asks for raw output.
- Keep safety/security detectors separate from product routing. Security filters may inspect text, but must not decide task decomposition or artifact delivery.
- Add tests proving equivalent behavior works without English-specific trigger words.
- Connected-account sources such as email, calendar, and payment-related records are opt-in per turn. Do not expose or call these tools unless the user explicitly asks to use that source in the current turn, or the turn is continuing a prior structured source selection.
- System prompts, developer prompts, model instructions, and user-facing copy may be written in English. The language-agnostic requirement applies to code paths that decide product behavior.
- Do not treat English prompt examples as dispatch rules. Prompts may explain the desired schema in English, but the runtime must branch only on typed model output, tool metadata, tool results, task/frame state, approval state, artifact descriptors, attachments, file extensions, URLs/domains, slash/operator commands, or other structured product signals.
- Do not fix language-coded behavior by expanding English phrase lists, adding translated synonym lists, or adding regexes for more wording. Replace phrase detectors with structured fields, deterministic tool-result summaries, explicit metadata, or a model-produced structured output validated against runtime evidence.
- If cleanup, retry, sanitizer, memory, or delivery behavior needs to know why a reply is incomplete, stale, ungrounded, missing a tool, asking a follow-up, or leaking internal state, carry that fact as structured metadata from the producer. Do not infer it from assistant prose such as "if you want", "not available", "I checked", "I found", or similar wording.
- When a model-produced plan or classifier is needed, ask for a language-neutral schema with enum fields and confidence/evidence, then validate the enum values and required runtime evidence before acting. Do not parse the model's explanatory prose.
- Activity/status summaries should show that a tool was used, including tools like `list_crons`, without exposing the tool's full output when that output contains internal task prose, paths, files, artifacts, credentials, connector payloads, or other non-deliverable state.
- User-facing tool results must pass through the shared structured evidence/presentation layer. Do not add per-tool or per-provider prose dump formatting. Concrete structured records, such as extracted page items, addresses, dates, prices, ratings, files, artifacts, emails, and calendar rows, must outrank generic source pages or transcript text when answering candidate, recommendation, lookup, or summary requests.
- Presentation, routing, and fulfillment code must not infer product behavior from English words inside user messages, snippets, page text, titles, or model prose. Language-specific text may be displayed after sanitization, but decisions must come from structured tool schemas, typed fields, URL/domain metadata, artifact descriptors, runtime state, approvals, or explicit model-produced structured plans.
- Durable user preferences that affect safety or consent, such as requiring confirmation before sending emails, must be saved as memory and injected into future turns across supported chat surfaces. Capture them through structured memory extraction or explicit memory tooling, not product-routing keyword checks.
- Any user-facing prompt that asks the user to choose from multiple options or provide one of several possible inputs must present numbered options and must allow a numeric reply such as `1`, `2`, or `3`. Do not require exact text names or internal ids as the only path for a normal user.
- Admin/operator requests to inspect or change Nullion itself should not dead-end when no direct config mutation tool is scoped. If structured admin principal/task state and a model-produced plan indicate a Nullion self-management change, offer the Web UI/browser fallback and ask for explicit confirmation before any UI mutation. After confirmation, use scoped browser/Web UI automation and verify the visible or runtime config before reporting success. Do not implement this with hardcoded phrase matching.
- Calendar connector capabilities must be derived from connector permission metadata and registered tool schemas. A write-capable calendar connector must expose the registered account-write calendar operations, such as `calendar_create`, `calendar_update`, `calendar_respond`, and `calendar_delete`, through structured tool scope behind approval boundaries. Do not tell the user the calendar is read-only when the provider is write-capable and those tools are registered.
- Browser sessions are workspace/lane state, not tab state. Reuse one persistent browser profile/session per workspace so cookies, login state, and storage survive across turns; open or switch tabs as needed without creating isolated browser sessions unless a structured isolation requirement exists.
- Ordinary browser navigation, website checks, and page-reading turns should default to navigation/text tools. Do not expose or call screenshot, image collection, media extraction, or visual-artifact tools just because a page has images; use them only when explicit structured task state, artifact/media requirements, attachment/image workflow, or the user's requested evidence requires visual capture.
- Tool selection must preserve the generic fallback chain. If a selected tool family only discovers candidates, previews, files, metadata, or a partial target, the next step must use the next viable structured capability or registered continuation tool before finalizing. Examples of generic fallback edges include search/list -> read/open, preview -> full body, browser navigation -> browser extraction/screenshot/click, file search -> file read/write when the user asked for file content or mutation, scheduler list -> scheduler run/update/delete when the user asked for an action, and local-system discovery -> terminal/computer-control approval flow when the user asked for a computer action. Do not replace these edges with prompt-specific wording or provider-specific hacks.
- A tool result that returns structured evidence must be answered from that evidence. Do not claim no result, ask the user for an id, or switch to saved history when a same-turn tool result already contains enough structured fields to answer or to call its registered continuation tool.
- A failed or insufficient tool path must not end the turn until adjacent safe fallback capabilities have been considered through structured metadata, tool continuation metadata, runtime permissions, or an explicit clarification. If an action is unsafe or requires approval, ask for approval; do not silently downgrade to a read-only listing.
- Nullion is a personal assistant for doing reasonable user-authorized work across the user's computer and connected accounts. For a new task shape that has no purpose-built tool, the generic fallback ladder is: use the safest dedicated structured tool first; if that only discovers a target, continue to its registered continuation; if no dedicated continuation exists, consider browser/computer-control/terminal capabilities with approval boundaries; if the next action is ambiguous or unsafe, ask a concise clarification or approval. Do not end at file/workspace/search/list/preview output when a normal user would expect the assistant to act on the computer.

## Anti-Use-Case Patching

- Never fix a reported bug by embedding the reporter's concrete prompt wording, nouns, account names, chat ids, repo names, branch names, product names, marketplace names, people names, country/language, screenshots, sample artifacts, URL path shapes, or vendor-specific semantics into Nullion core behavior.
- Do not implement behavior that only passes the user's current test prompt, current screenshot, current account, current connector, current website, current cron, current branch, current artifact name, or current QA fixture. A fix is not acceptable unless it works for a different user, different workspace, different wording, different channel, and different but equivalent structured runtime facts.
- Treat every concrete report as an example of a product invariant, not as the thing to match. First state the invariant in implementation notes, then implement that invariant with typed runtime evidence such as tool schemas/results, URL/domain records as opaque references, attachment metadata, artifact descriptors, task/frame state, approval state, stored memory records, or model-produced structured plans.
- Core may preserve opaque structured evidence, like `{"type": "url", "domain": "...", "url": "..."}` or tool names/result status, but core must not interpret provider-specific URL paths, repo semantics, marketplace semantics, order semantics, or connector defaults. Put provider behavior in the connector, adapter, skill pack, or tool metadata that owns it.
- When multiple saved-history, memory, tool, artifact, or URL candidates exist, do not choose by old rank, English wording, or the user's previous example. Use current structured evidence and live tools when available; otherwise ask a brief clarification.
- Search, memory, and recent-context fixes must be language- and user-agnostic. Use corpus statistics, recency, typed evidence, and structured relationships instead of English trigger words, regex synonym lists, or one user's recurring topics.
- Tests for these fixes must use neutral synthetic fixtures and include at least one wording or language variant that does not depend on the original English example. Do not use the reporter's actual accounts, products, stores, repos, branches, artifact names, cron names, or chat ids as the thing that proves core behavior.
- Before handoff for any bug fix, audit the touched app diff for reporter-specific, QA-specific, branch-specific, stage/prod-specific, prompt-specific, and vendor-specific strings. Remove them from core behavior, or explicitly justify each remaining provider name as adapter/UI copy, documentation, connector metadata, or test fixture data rather than routing logic.
- If a live test requires concrete names, URLs, screenshots, files, accounts, or prompts, those values are evidence only. They must never become dispatch keys, routing rules, output formatting rules, retry rules, delivery rules, connector defaults, or special cases in Nullion core.
- Any fix involving one tool must audit adjacent tool-family paths for the same failure mode before handoff: discovery-only answers, missing continuation tools, raw structured dumps, wrong saved-history fallback, wrong reply anchor, repeated identical failures, permission/approval downgrades, artifact sidecars, and platform formatting. Add regression coverage for the invariant, not just the reported prompt.
- Every user-reported live tool fallback bug must get a live stage regression scenario before the fix is called ready. The scenario must cover the original affected surface and at least one adjacent or follow-up variant that proves the generic fallback chain, not the exact reported wording.

## Product And Install Truth

- Nullion is an open-source product intended for millions of users running it in their own ways, on their own machines, with their own operating systems, accounts, files, connectors, providers, dependency caches, permissions, and workflows. Do not treat this developer machine, a stage/prod venv, a local global package, a personal account, or a one-off live run as the product contract.
- The source of truth for shipped Python dependencies is the repository's declared install metadata, including `pyproject.toml`, lockfiles, installer scripts, updater dependency logic, capability dependency catalogs, and packaged skill/connector metadata. Before assuming a library, CLI, browser, connector, or binary exists for users, verify it is declared in the shipped install path.
- If a fix needs a new runtime dependency, add it deliberately to the product dependency contract and every relevant install/update/capability surface. Do not silently rely on packages that happen to be installed in `~/.nullion`, `~/.nullion-stage`, the current shell, Homebrew, npm globals, or any local virtual environment.
- If a declared required dependency is missing at runtime, treat that as an install/update/runtime health issue with an actionable error or repair path. Do not hide a broken installation by adding a degraded local workaround unless the fallback is a deliberate, tested product feature that works in a clean install.
- Product fixes must be validated against a clean-install mental model: fresh checkout, declared dependencies only, no user-specific absolute paths, no private files, no local caches, no manually installed helper tools, no stage-only credentials, and no assumptions about the reporter's machine.
- Live stage/prod evidence is valuable for proving behavior, but it is not permission to encode that environment into core behavior. Use live evidence to discover the invariant, then implement the invariant through shipped metadata, typed runtime state, tool schemas/results, artifact descriptors, approval state, or adapter-owned configuration.
- When comparing Nullion with another agentic app or open-source project, extract durable product patterns such as typed artifacts, verified delivery receipts, dependency declarations, and continuation contracts. Do not copy anecdotal fallback behavior unless it is compatible with Nullion's shipped install contract and tested on a clean dependency surface.
- Before handoff for dependency, artifact, tool, connector, browser, archive, document, spreadsheet, media, installer, or updater fixes, explicitly check whether the change depends on anything outside the shipped product contract. Remove the assumption or update the contract before claiming the fix is product-ready.

## User-Facing Output Quality

- This is a strict product contract for every chat surface, cron delivery, planner result, tool result summary, artifact caption, approval prompt, error message, and operator command response: output sent to a user must be readable, structured, and intentionally formatted. Do not ship raw model notes, raw tool output, transcript fragments, JSON/Python dicts, internal state, stack traces, local paths, connector payloads, prompt text, or collapsed wall-of-text lists as the user-visible answer.
- User preferences and explicit requested format always supersede the default presentation style. If the user asks for raw JSON, terse output, prose, a table, Markdown, numbered steps, or another concrete format, honor that format as far as possible while still protecting credentials, secrets, private payload bodies, local paths, and internal-only runtime state.
- Default final replies must start with the concrete answer or outcome, then group supporting details under short human labels such as `Summary`, `Done`, `Blocked`, `Files`, `Why`, `Next`, or numbered choices. Do not make users parse a paragraph to discover the result.
- Lists must render as real lists. If a model produces inline numbered markers such as `1. ... 2. ... 3. ...` or `(1) ... (2) ...`, normalize them into separate lines before delivery. If the output contains steps, options, acceptance criteria, findings, files, or multiple actions, preserve line breaks and list structure across Telegram, Web, Slack, Discord, and cron delivery.
- Planner cards and activity cards are status surfaces, not substitutes for final answer quality. Do not change planner-card semantics to hide an unreadable final answer. Fix the final-reply formatting path, shared platform delivery path, or response sanitizer so the same readable output contract holds on every channel.
- Tool and connector outputs must be summarized from structured fields into user-facing language. It is acceptable to mention tool names only when useful for debugging, requested by the user, or needed to prove execution; never expose full tool results, connector inventories, prompt/tool schemas, credential refs, message bodies, calendar bodies, file system paths, or internal task prose by default.
- Error and blocked-state replies must be concise and actionable: state what failed, the user-visible impact, and the next safe action. Do not leak implementation details or retry logs unless the user explicitly asks for diagnostics.
- Any change affecting reply generation, sanitizer behavior, planner final output, activity/status delivery, artifact captions, cron delivery text, or messaging adapters must include tests proving readable final output on the shared path. For user-visible channel bugs, verify through the real affected surface when authorized; helper-only tests are not enough.

## Planner Cards And Activity Display

- Planner card visibility is controlled only by the settings-backed planner/task-card setting, such as `NULLION_TASK_PLANNER_FEED_MODE` and the matching UI setting, not by `/verbose`.
- `/verbose` is only `on`, `off`, or `status`; it controls whether activity details are shown. Do not reintroduce planner/full verbose modes or a public task-card command.
- When planner cards are enabled, every planned run should render a planner card on supported surfaces, including Web, Telegram, Slack, and Discord. The card should include `PLANNER`, the mission type, a compact `For: <request summary>` line from typed task-status state, the task count/title, and per-task rows.
- When activity feed is off, show only the planner card top section: header, request summary, task count/title, and task rows. Do not append the `ACTIVITY` section or tool/mini-agent details below it.
- When activity feed is on, show the same planner card top section plus an `ACTIVITY LIVE` section below it with compact, deduped tool/mini-agent activity.
- Task-status delivery must pass and branch on typed state such as `status_kind`, `group_id`, and, where applicable, `include_activity`; clients must not infer planner-card or activity-feed behavior from user prompt prose or stale local UI state.
- Terminal success, failure, timeout, or artifact completion must always deliver a final user-visible result message or verified artifact receipt. Never suppress terminal results just because a planner card or activity card was visible.
- For `TaskDecomposer` and planner-preview latency regressions, debug and fix the active provider/runtime bottleneck first. Do not implement model-specific routing or provider-specific workaround behavior as the primary fix.
- Any decomposer slowness investigation must capture runtime evidence before UI/display patches: `cron planner preview profile` timing logs, `TaskDecomposer` timeout/parse errors, prompt size, tool-count, and `list_crons -> run_cron` startup gap from `runtime.db`.
- Do not assume users have a second provider or a "fast lane" model. The default path must remain reliable when only one provider/model is configured.
- If fallback planner cards are needed, they are resilience-only and must not replace root-cause performance fixes.
- Add regression tests that cover timeout, clarification, and valid multi-task planner outcomes without relying on English trigger words.
- Latency is a release-critical product requirement. Do not fix routing, scheduler, connector, planner, or activity bugs by adding a model/classifier call to every ordinary request. First look for typed runtime signals, cached decisions with correct invalidation, existing task/frame state, tool result schemas, stored job/artifact descriptors, and other cheap structured routes. If a new model decision is unavoidable, gate it behind structured evidence and add tests proving unrelated plain turns stay on the fast path.
- Startup should proactively warm cheap, reusable runtime facts that chat turns commonly need, including tool registry shape, cron metadata, compact installed-skill indexes, system context, and platform chat context. Do not move this work into the hot path for every ordinary chat request unless the cache is missing or invalid.
- Tool scoping must follow this order: use the cached compact installed-skill/capability index for cheap awareness, make a structured scope decision or `request_tool_scope` call only when the visible tools are insufficient, then load exact skill docs or connector context only after that structured scope is selected. Never load full skill-pack text, connector inventories, or scheduler/job details on every simple chat turn.
- Do not hardcode product behavior, URL rewrites, endpoint defaults, workflow assumptions, retries, or output fixes for third-party skill packs, connector providers, external APIs, or vendor-specific routes inside Nullion core. Core may enforce generic safety, schema, permission, boundary, delivery, and tool-contract rules only. If a third-party skill pack or connector returns the wrong default behavior, fix the installed skill pack, connector metadata, adapter configuration, or skill documentation instead of encoding that provider's semantics in Nullion core.
- Local shell fallback is a real computer-control path, not a file-search substitute. Read-only diagnostic or no-change turns must not run shell commands that mutate files, symlinks, package state, services, app settings, or other local system state. Any terminal command that would write/delete/modify local state outside the trusted workspace must be blocked or routed through structured approval/boundary facts before execution, independent of the natural-language wording that led to the tool call.

## Verification Guardrails

- For live behavior reports from prod, stage, dev, Telegram, Web, Slack, Discord, or cron delivery, inspect runtime evidence before guessing or patching: relevant logs, `runtime.db` events/tool timings, process/import path, active lane, provider/connector status, and exact user-visible error text. Do not diagnose from screenshots or assumptions alone when logs/runtime state are available.
- For user-visible delivery bugs, verify through the live stage Web UI or live stage Telegram app platform boundary after patching, not only helper tests or HTTP/API probes. Reuse the same prompt shape and surface that failed when the user authorizes it.
- For chat-surface bugs, automated `/api/chat` probes are diagnostics only. They may support investigation, but they are not a live-stage pass condition and must not be used as a shortcut for Web UI or Telegram app validation.
- If a user asks to test with the Telegram app and Web app, do exactly that before claiming the fix is tested: drive the stage Web UI in a browser and wait for a real inbound user message in the stage Telegram app. Do not substitute `/api/chat`, direct runtime calls, synthetic updates, bot-token messages, database inspection, or logs for either surface; those are evidence only after the actual surface was exercised.
- For user-visible chat-surface bugs, "live tested" means both supported live chat surfaces that can exercise the failure: the actual stage Web UI in a browser and the actual stage Telegram app/user-message path. Do not claim a chat-routing, reply-context, status-card, tool-scope, attachment, artifact, formatting, latency, or follow-up fix is live-tested after only `/api/chat`, helper/unit tests, handler-only tests, bot-token-only sends, runtime DB checks, logs, or one surface. If the failure can appear in both Web and Telegram, run both. If one surface cannot be tested, state the exact blocker and do not call the branch fully live-tested.
- For Telegram reply-to, quoted-message, numeric-choice, follow-up, interruption, or concurrent-task bugs, live validation must include a real user-sent Telegram app message using the same platform action that failed, such as replying to a bot message in the Telegram UI. Bot-token sends, synthetic updates, and direct runtime/API calls may be useful coverage, but they do not prove the real Telegram app reply-to path.
- For Web UI bugs that affect chat, history, settings, artifact delivery, or browser-readable state, live validation must include the actual Web UI in a browser, not only HTTP responses. Verify the visible UI state and the backing runtime evidence when possible.
- Every user-reported regression that is fixed must add or update a repeatable stage/live regression check in `nullion-test` when feasible. The test should exercise the product boundary that failed and include the relevant structured evidence, such as reply metadata, selected conversation id, scoped tools, delivered artifacts, or visible UI text. If a fully automated live test is impossible because it needs a real Telegram user action, add the closest automated coverage and record the required manual live Telegram step in the QA notes/final handoff.
- For Telegram-facing changes, do not count `/api/chat`, helper tests, direct runtime calls, synthetic updates, or bot-token-only Bot API sends as full Telegram validation. A bot token can send messages as the bot but cannot impersonate the user's inbound Telegram account, so it does not prove the real inbound user-message path. Full Telegram validation requires a real user-sent Telegram message in the stage Telegram app on the target lane while the agent watches the matching Telegram logs, `runtime.db` turn/tool/artifact records, and user-visible delivery outcome. If a fake `Update` or handler test is used, label it clearly as handler-only coverage and still request/perform live user-message verification before claiming Telegram is tested.
- For tool-scope, connector, media-generation, scheduler, weather, or artifact-delivery regressions, record the exact failed turn before guessing. The failure chain must include the selected surface, conversation id or request id when available, scoped tool list/tool count, tool results, artifacts, and timing.
- Artifact delivery regressions must be tested against prior fixed cases before handoff: generated cron reports, email/send receipts with attachments, platform captions, and internal sidecar/state-file suppression.
- The active pre-commit hook is `.githooks/pre-commit`, selected by `core.hooksPath=.githooks`.
- The app pre-commit hook blocks `test/`, `tests/`, `test_*.py`, and `*_test.py` from this repo.
- The app pre-commit hook runs the private suite in `../nullion-test/tests` with coverage unless `NULLION_SKIP_PRIVATE_TESTS=1` is set.
- Every bug fix must include the right test layers before it is marked ready: a focused regression test for the changed code path and, when the bug affected a user-visible workflow, an E2E test in `nullion-test` proving the real product boundary that failed.
- Integration tests may supplement the E2E, but they do not replace it for user-visible workflow bugs unless an E2E cannot be written.
- If a required E2E cannot be written, record the concrete reason in the QA tracker and final handoff instead of silently skipping it.
- While implementing, run focused tests for the touched path only. Do not run the full private suite, all tests, or broad E2E matrix until all code, test, and tracker updates for the current request are complete and targeted checks are green.
- After all work for the current request is complete, run the broadest relevant local verification once, or report the exact blocker before push, merge, or final handoff.

When fixing a latency-sensitive chat, routing, connector, scheduler, planner, or activity path, add or update an E2E that runs multiple representative requests, records elapsed time, asserts an average/p95-style threshold, and simulates the previously slow dependency so the test fails if that dependency re-enters the hot path.

Before handoff, run focused local tests and then verify the user-visible behavior on live stage Web UI and/or live stage Telegram app, matching the surface that failed.
